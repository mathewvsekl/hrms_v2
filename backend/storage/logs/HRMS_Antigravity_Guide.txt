PART 1: WHAT IS GOOGLE ANTIGRAVITY?

Google Antigravity (released November 2025) is Google's free, agent-first development platform. Unlike traditional IDEs that offer autocomplete, Antigravity lets autonomous AI agents plan, write code, run terminal commands, test in a browser, and validate results — all with minimal manual intervention. It is the perfect IDE for your HRMS blueprint because your blueprint already defines 8 named AI agents (Eva, Liam, Sofia, etc.). Antigravity's architecture was designed for exactly this kind of multi-agent, parallel development workflow.

1.1 Two Core Surfaces

SurfaceWhat it does & when to use itEditor ViewFamiliar VS Code-style IDE with an agent panel on the side. Use for hands-on coding, reviewing agent-generated files, inline commands, and tab completions. Ideal for Phase 1 setup work.Agent Manager (Mission Control)Spawn and orchestrate MULTIPLE agents working in PARALLEL across different workspaces. Shows live status, Artifacts, and Inbox for approvals. This is where your 8 HRMS agents (Eva, Liam, Sofia, Noah, Maya, Aria, Oscar, Nova) will operate simultaneously.

Nova will use Artifacts to report to the human supervisor before deployment.RulesPersistent 'system instructions' always active for every agent session. Use these to enforce RBAC rules, coding standards, database conventions, and HRMS business logic.WorkflowsSaved prompts triggered with /command. PART 4: CONFIGURE ANTIGRAVITY FOR YOUR HRMS
This is the most important Antigravity-specific section. Properly configured Rules, Workflows, and Skills mean your agents will follow HRMS business logic automatically — without you having to repeat instructions every session.

Step 6 — Create Global Rules
Rules are always-on system instructions injected into every agent session. Create .antigravity/rules/global.md with these HRMS rules:

# HRMS Global Rules (always active for all agents)
- ALWAYS store datetime values in UTC. Never store local times in the database.
- ALWAYS use JSONB (not JSON) for custom field columns in PostgreSQL.
- NEVER hard-code currency codes, timezones, or country rules — read from offices table.
- EVERY database write must trigger an Oscar audit log entry.
- RBAC must be enforced on every API endpoint via FastAPI dependency injection.
- Agents MUST operate only within their assigned module scope (defined in RBAC config).
- All agent actions are sandboxed — no direct writes to production DB without Nova approval.
- Use Plan Mode for any task touching payroll or employee data.
- Each function must have a docstring describing its RBAC scope.
Step 7 — Create HRMS Workflows
Workflows are /command shortcuts you trigger on demand. Create these in .antigravity/workflows/:

Workflow /commandWhat it does/build-module [name]Scaffold a new HRMS module (routes, models, schemas, tests) following project conventions/run-simulationTrigger Maya to generate synthetic employee/payroll/attendance data for testing/verify-rbacTrigger Noah to validate all RBAC permissions match the blueprint specification/generate-audit-reportTrigger Oscar to produce an audit trail report for the last 7 days/check-integration [module]Trigger Aria to test cross-module workflow for specified module/supervisor-reportTrigger Nova to compile all agent outputs into a deployment readiness report/run-payroll [office_id]Trigger Eva to simulate payroll calculation for a specific office/offboarding-check [emp_id]Run full offboarding checklist: final payroll, access revocation, audit
Step 8 — Create Agent Skills
Skills are modular knowledge packages that load only when relevant — preventing context bloat. Create one skill per HRMS domain:

PART 10: MULTI-OFFICE, MULTI-COUNTRY, MULTI-CURRENCY

Step 18 — Timezone Architecture

Critical Rule: ALL datetimes stored as UTC. Convert to local time only at the display layer.
• Attendance check-in: client sends ISO 8601 with offset ? backend converts to UTC ? stores UTC + original offset
• Payroll cut-off dates: stored as UTC, presented in office local time on frontend
• Leave day counting: use office timezone for date boundaries, query in UTC
• Liam's workspace Rule: add to .antigravity/rules/attendance.md — 'Never store local times in DB'

Step 19 — Multi-Currency Payroll
• Store all salary with ISO 4217 currency code — never store as plain numbers without currency
• Each office has its own payroll cycle in its own currency — Eva never converts between currencies
• Super Admin consolidated report: fetch live FX rates via external API, store as snapshot alongside report
• Eva's workspace Rule: 'Always include currency_code in every payroll calculation output'

Step 20 — Office-Level Custom Fields
This is the most architecturally unique part of your system. Implementation flow:

24. Super Admin defines a custom field template for an office — stored as JSONB schema in offices.custom_field_templates
25. When an employee is assigned to that office, their employee_offices record gets an empty custom_field_values JSONB object matching the template
26. Sofia validates every custom field write against the office's template schema before saving
27. Frontend dynamically renders custom field forms by reading the office template schema
28. All HRMS module tables (payroll, attendance, leave, appraisal) have their own custom_field_values JSONB column
